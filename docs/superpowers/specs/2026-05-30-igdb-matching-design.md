# IGDB Matching Algorithm — Design Spec

**Date:** 2026-05-30  
**Status:** Approved

## Problem

The current IGDB matching pipeline generates too many items in the review queue AND silently auto-confirms wrong matches. Two root causes:

1. `name ~ "X"` is a CONTAINS operator on IGDB — 1 result is treated as confidence 1.0 but could be a different game
2. When multiple results return, the shortest name is auto-picked (bad heuristic) with no review
3. The fuzzy confidence range 0.5–0.7 is too broad — too many games fall here
4. Review UI shows only one proposal with no alternatives to choose from

## Goals

- **A** — Reduce the review queue to the minimum: only genuinely ambiguous cases pass through
- **B** — When a game does reach review, show multiple IGDB candidates to choose from (1-click confirm)

## Algorithm — Single-Pass with Alternative Names

### Replace 3-attempt pipeline with 1 IGDB query

**Old flow:** `name ~` exact → name variants loop → `search` fuzzy (2–3+ API calls)

**New flow:** Single query per game:

```
fields id, name, platforms, alternative_names.name;
search "{cleaned_name}";
where platforms = ({platformId});
limit 10;
```

The IGDB `search` operator already searches across main names and alternative names. Fetching `alternative_names.name` lets us score locally against regional/translated titles (e.g. "Akumajō Dracula" → "Castlevania").

### Scoring

For each IGDB candidate returned:

1. Compute Levenshtein similarity between the cleaned ROM name and the candidate's `name`
2. Compute similarity against each `alternative_names[].name`
3. Also compute against local ROM name variants (roman numeral conversions, accent removal, pre-colon shortening) generated by `generateNameVariants()`
4. **Candidate score = max of all above comparisons**

Sort all candidates by score descending. The top candidate determines the outcome.

### Confidence Thresholds

| Score | Action |
|-------|--------|
| ≥ 0.92 | Auto-confirm, `needsReview: false` |
| 0.65–0.91 | Flag for review, `needsReview: true` |
| < 0.65 | `not_found`, `needsReview: false` |

When no platform mapping exists, multiply the final score by 0.7 (existing behaviour) and always set `needsReview: true` if a match is found.

### Match Methods

- `exact_name` — top candidate score ≥ 0.92 via main name match
- `alt_name` — top candidate score ≥ 0.92 via an alternative name match
- `fuzzy` — top candidate score 0.65–0.91
- `not_found` — top candidate score < 0.65 or no results

## Candidates Storage

Add a `candidates` column (JSON, nullable) to `gameIgdbMapping`.

**Schema addition (Drizzle):**
```ts
candidates: text('candidates') // JSON: IgdbCandidate[]
```

**Type:**
```ts
type IgdbCandidate = {
  igdbId: number
  igdbName: string
  confidence: number
}
```

- Store **top-5 candidates** on every match (including auto-confirms)
- The selected/auto-confirmed game remains in `igdbId` / `igdbName` columns as before
- `candidates` is purely for the review UI and future corrections

**Migration:** Drizzle `generate` + `migrate` (additive column, no data loss).

## Review UI — Multi-Candidate Selection

### Current state
Single proposal (ROM name + IGDB name), actions: Confirm / Reject / Manual entry.

### New state

When `needsReview: true`, display:

1. **ROM source name** (header)
2. **Primary candidate** (candidate[0]) — visually prominent, with confidence badge
3. **Candidates 2–5** — listed below, each with IGDB name + confidence score + "Select" button
4. **"None / not found"** button (= current reject)
5. **Manual entry** (IGDB ID input) — collapsed by default, expandable

### API changes

`GET /api/igdb/review` — include `candidates` field in each item (parsed from JSON column).

`POST /api/igdb/review/confirm` — no schema change needed. Selecting any candidate uses existing `action: "manual"` with `igdbId` + `igdbName`. Confirming the primary proposal uses `action: "confirm"`.

No IGDB cover thumbnails in this iteration — adds a separate `/covers` API call with marginal UX gain.

## Files to Change

| File | Change |
|------|--------|
| `lib/igdb/match-game.ts` | Rewrite core algorithm: single query, alt_names scoring, new thresholds |

| Review UI component | Show multi-candidate selection |
| Drizzle migration | Generate + apply migration |

The `clean-rom-name.ts` and `batch-match.ts` files require no changes. `match-single.ts` (fire-and-forget) uses `matchGameToIgdb` directly — benefits automatically.

## Out of Scope

- IGDB cover thumbnails in review UI (future iteration)
- Year-based disambiguation from ROM filename
- `first_release_date` field from IGDB
- Re-matching already confirmed games
